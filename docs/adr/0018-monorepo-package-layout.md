# Monorepo package layout: four apps, six packages, with domain as a pure package

The repo uses Bun workspaces + Turborepo. Apps live in `apps/` (`api`, `marketplace-web`, `business-web`, `admin-web`) and shared code lives in `packages/`: `db` (Drizzle schema, migrations, query helpers), `domain` (pure domain rules — Booking lifecycle, deposit forfeit, NoShowRisk computation, BookingIntent reassignment — no I/O), `contracts` (auto-generated API contract types consumed by all web apps), `ui` (design-system primitives only — Button, Input, Modal, Form — never feature components), `validation` (Zod schemas shared by api and web for one-source-of-truth input validation), and `config` (shared tsconfig / eslint / prettier). The load-bearing decisions are: (1) `domain` is a separate package with zero I/O dependencies, so booking rules are unit-testable without a database or HTTP server and cannot accidentally couple to Drizzle; (2) `contracts` is generated from the API at build time, not hand-maintained, to prevent the silent type drift that hand-maintained contract packages always suffer; (3) `ui` is strictly primitives — feature-level components live in their owning app, because marketplace / business / admin have different visual idioms; (4) `validation` is shared client+server so the same schema gates both sides. No `i18n` package in v1 (French only — see ADR-0016); add later when Arabic lands.
