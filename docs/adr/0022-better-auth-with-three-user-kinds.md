# Auth: Better Auth, single User table with kind discriminator, DB-backed sessions

Authentication uses Better Auth across all three contexts on a single User table discriminated by `kind` (`client` | `staff` | `admin`). Login methods per kind: Client = phone + OTP only (per ADR-0007); Staff (Business owner, Practitioner, receptionist) = email + password, with phone OTP available as fallback / second factor; Admin = email + password + mandatory TOTP MFA, provisioned manually with no public signup route. Sessions are database-backed (Drizzle-managed) rather than stateless JWTs — the cost of one query per request is well worth the ability to revoke on staff termination, fraud, or dispute. Within Business context we ship a minimal role model in v1: `owner` (full access), `practitioner` (own schedule and own bookings), and a placeholder `member` to be refined when customer feedback gives us real signal. We rejected Clerk (US-centric, billing model fits poorly), Supabase Auth (couples us to Supabase as a hosting choice), and Lucia (its 2026 "build your own" stance adds week-one velocity cost we don't need to pay).
