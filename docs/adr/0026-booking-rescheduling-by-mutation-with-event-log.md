# Rescheduling mutates the Booking; an immutable event log is the audit trail

A reschedule updates the Booking's time fields in place rather than creating a new Booking. Identity fields (Booking ID, Client, ServiceOffering) are never rewritten; only `startsAt`, `endsAt`, and (if reassigned) `practitionerId` are mutable. Every change writes a `BookingHistoryEvent` row — created, rescheduled, practitioner-reassigned, cancelled, marked-no-show, auto-completed — capturing actor, role, time, and payload. The event log is immutable; it is the source for the audit trail, analytics, and dispute resolution. We rejected cancel + new-Booking because the deposit refund-and-recharge cycle through Konnect (1–3 day delay, two operations, partial-failure modes) made reschedule operationally hostile, and the only way to avoid it was to introduce a special "deposit transfers" rule that re-introduced the complexity that mutation already solves. Permission rules: Client can reschedule within the cancellation cutoff; past the cutoff they must cancel (forfeit Deposit) and rebook. Business can reschedule any time, but past-cutoff Business reschedules require Client confirmation via a `pending-reschedule` flag-state with a TTL. Slot-conflict integrity on the new time is guaranteed by the same Postgres EXCLUDE constraint as Booking creation (per ADR-0023).
