# Practitioner-Venue is many-to-many via Schedule blocks; Booking inherits Venue from the schedule

A Practitioner has a single calendar (per ADR-0002) and a Schedule made up of working blocks; each block is tagged with one Venue (start, end, breaks). A Practitioner can work across multiple Venues of the same Business by having blocks at different Venues — no double-booking is possible because the calendar is one. A Booking is created `(Practitioner, ServiceOffering, time)`; the Venue is derived from the Schedule block the time falls inside, never stored separately on the Booking. We rejected one-Venue-per-Practitioner (forced hacky duplicate Practitioner rows for chains with shared staff) and many-Venues-without-schedule (no constraint, allows nonsense like "Sophia booked at Venue A at the same time she's working at Venue B"). The Booking discovery UX supports three first-class entry points (Practitioner-first, Venue-first, ServiceType+location search); the data model is the same in all three. Reassignment policy: Practitioner can be silently swapped only if BookingIntent is `practitioner-flexible` *and* the new Practitioner is at the same Venue at the same time; Venue can never be changed silently — always requires Client consent.
