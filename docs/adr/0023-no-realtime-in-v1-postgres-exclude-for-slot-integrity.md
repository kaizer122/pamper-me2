# No real-time transport in v1; slot integrity guarded by a Postgres EXCLUDE constraint

The v1 stack has no WebSocket or SSE transport. Business calendars and Bookings inboxes refresh by polling the API every ~15s. Real-time correctness on the Booking write path is enforced at the database level, not in transport: a Postgres `EXCLUDE` constraint using `tsrange` overlap with `gist` on `(practitioner_id, occupied_range)` makes double-booking the same Practitioner physically impossible. The API write path uses a transaction with row-level locking on the Practitioner Schedule for the relevant window; concurrent Booking attempts collide cleanly with the constraint, and the API translates the violation into a friendly "this slot was just taken" response. We chose this over WebSockets (real ops cost: connection state, reconnection, sticky sessions / pubsub broker, scaling) and SSE (lightweight but still infrastructure we can defer) and managed pubsub (Pusher/Ably — vendor cost we don't need yet). v1.5 is expected to add SSE for Business calendar live updates; that change is purely additive — events emit from the same code path that already writes Bookings, so the data model doesn't move. A future engineer should not solve double-booking at the application layer with locks or queues — the EXCLUDE constraint is the load-bearing guarantee.
