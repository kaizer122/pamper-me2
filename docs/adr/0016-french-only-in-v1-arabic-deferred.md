# French is the only supported language in v1; Arabic deferred

The platform ships in French only for v1. No translation infrastructure, no per-field locale storage, no RTL layout work — text fields are single-column French values throughout the schema and UI. We considered strict bilingual (FR/AR required everywhere) and per-field optional bilingual with fallback, but rejected both for v1 because translation infrastructure adds meaningful frontend, content-pipeline, and Business-onboarding cost without being on the critical path for shipping. We are explicitly accepting that this excludes the Arabic-first segment of the Tunis market — a meaningful slice, especially in older and lower-French-fluency demographics that beauty and wellness do serve. Adding Arabic later is a deliberate v2 workstream that requires: schema changes (translation tables or per-field locale columns), RTL frontend audit (logical CSS properties, `dir` attribute, mirrored icons), bilingual ServiceType catalog, and bilingual platform-controlled content (system emails, error messages). A future engineer should not assume Arabic was an oversight — it was deferred.
