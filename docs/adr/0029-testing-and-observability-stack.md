# Testing: bun test + Vitest + Playwright. Observability: Sentry + Pino + OTel

Testing splits across three layers. `bun test` (native, fast) runs unit and integration tests for `packages/domain`, `packages/db`, and `apps/api` — `api` integration tests run against a real test Postgres via Testcontainers or a docker-compose fixture (never against mocks, because the EXCLUDE-constraint slot integrity guarantee is part of the system under test). Vitest + Testing Library handles component / DOM tests in the three web apps; chosen over `bun test` because Vitest's React tooling is more mature in 2026. Playwright handles E2E with one smoke suite per app plus a cross-app booking flow that exercises Marketplace → API → Business calendar in one run. Traditional contract tests are unnecessary because `packages/contracts` is auto-generated and type errors at build time replace runtime contract verification.

Observability: errors go to Sentry (Bun-integrated SDK in all four apps; sourcemap upload in CI); structured logs use Pino on `apps/api` (JSON output); metrics and traces are emitted via the OpenTelemetry SDK from `apps/api` from day one — the OTel *backend* (Honeycomb / Tempo / Datadog) is intentionally deferred until something is running, but instrumentation is not. Health-check pings via Better Stack hit `/healthz` on each app. The technical observability stack is for signal on technical concerns only; the domain audit trail is `BookingHistoryEvent` (per ADR-0026), not log scraping.
