# Postgres with Drizzle as the only data layer; migrations are committed SQL

The persistence layer is Postgres accessed via Drizzle, exposed to the rest of the codebase through `packages/db` (schema, query helpers, repository functions that translate rows to domain shapes). The `domain` package has zero database imports — rules are pure and testable without a database; only `api` composes `domain` and `db` together. We chose Drizzle over Prisma (heavier runtime, separate codegen DSL, weaker on Bun), Kysely (no migrations story, would need bolting on), and raw SQL (maintenance cost grows fast). Migrations: Drizzle Kit generates SQL files from schema diffs; the SQL files are committed to the repo; CI applies them in order against the deploy target before promoting traffic. We never `db push` against production — the SQL audit trail is non-negotiable for a payments-adjacent system.
