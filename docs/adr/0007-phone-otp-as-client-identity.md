# Client identity is anchored on a phone number verified by OTP; password is optional

A Client's canonical identity is a phone number verified via OTP at first booking. Setting a password is optional and happens after the first booking, not before. This collapses the "guest vs registered" distinction: a so-called guest Client is just a Client without a password yet, not a separate entity. We rejected an email-as-identity model because beauty bookings already require a working phone for confirmations and reminders, and email-based merging across booking attempts is a fraud and bug magnet (typos, plus-aliasing, gmail dot tricks). OTP verification at booking is therefore non-negotiable for marketplace-channel Bookings: removing it would force us back to a separate PendingClient / merge model that is materially harder to get right.

**Scope refined by ADR-0027:** the OTP-required rule applies only to marketplace-channel Bookings. Walk-in and phone-channel Bookings (Business-created) may attach to an unverified Client; an `identityVerified` flag distinguishes the two and unverified Clients merge into verified ones when the same phone later completes OTP via the marketplace.
